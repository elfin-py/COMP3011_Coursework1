\section{Overview}
Style Forecast is an API-first outfit recommendation system designed for a practical everyday problem: people often have limited time when getting ready, incomplete awareness of weather conditions at a chosen time, and only partial knowledge of current style trends. The system bridges that gap by combining live weather context, magazine-derived trend signals, and lightweight user context into a recommendation that helps the user feel both comfortable and confident.

The central design objective was not simply to return an outfit, but to return one that is appropriate for climate, relevant to current fashion context, and explainable enough to earn user trust. This emphasis on explainability shaped both the system architecture and the recommendation logic, and it differentiates the project from many fashion tools that prioritise inspiration or browsing over defensible recommendation behaviour.

\section{Methodology}
\subsection{Problem Framing and Design Approach}
The project was approached as a decision-support system rather than a generic fashion catalogue. A recommendation is only useful if it reduces uncertainty at the point of decision-making, so the system needs to reason about context rather than preference alone. For that reason, the chosen methodology combines:
\begin{itemize}
  \item \textbf{weather suitability}, to reduce the risk of physical discomfort,
  \item \textbf{trend relevance}, to reflect recent fashion signals,
  \item \textbf{bounded scoring}, to keep the recommendation stable, explainable, and technically defensible.
\end{itemize}

Alternative approaches were considered before selecting the final model. Pure collaborative filtering was not suitable because the dataset is small and suffers from cold-start problems. An end-to-end LLM ranking approach was also rejected because it would be harder to audit, less deterministic, and less reliable for a live assessment setting. GraphQL was considered as an alternative API style, but the assessed workflows favour stable resources, explicit status-code behaviour, and straightforward Swagger documentation. A hybrid weighted model therefore offered the best balance of interpretability, control, and practical usefulness.

Style Forecast is not presented as the first system to combine fashion and weather cues. Its originality lies in how it integrates forecast context, magazine-derived trend signals, and bounded recommendation logic into a single explainable pipeline that can be defended technically. In practice, a user could try to recreate a similar decision process manually with a general-purpose AI assistant such as ChatGPT, but that would require repeatedly gathering weather data, checking current fashion coverage, restating context, and interpreting unstructured responses each day. Style Forecast streamlines that fragmented task into a focused, repeatable API workflow with explicit inputs, auditable scoring, and an interface tailored to this specific decision.

\subsection{Technology Choices and Rationale}
\textbf{TypeScript} was chosen across frontend and backend to reduce integration mismatch and improve refactor safety \cite{typescript}. Using one typed language simplified DTO design, API contracts, and debugging.

\textbf{NestJS} was chosen for the backend because its modular structure fits a system with distinct domains such as authentication, climate retrieval, trend ingestion, persistence, and recommendation logic \cite{nestjs}. Keeping logic in services rather than controllers improved clarity and made it easier to adjust scoring logic without destabilising routing behaviour.

\textbf{PostgreSQL} with \textbf{Prisma} was chosen for persistence \cite{postgresql,prisma}. An SQL database is appropriate because the project still has relational concerns: users, saved recommendation snapshots, climate records, and trend entries all benefit from structured querying and clear relationships. The schema also includes item and outfit resources used to satisfy the assessed CRUD requirement and support stored recommendation structures. Figure~\ref{fig:data} summarises the core persistence model used by the app and backend services.

\textbf{React + Vite} was chosen for the client because the interaction model is stateful and iterative \cite{react,vite}. React supports interactive recommendation flows and authenticated UI state, while Vite keeps iteration fast.

\textbf{REST over GraphQL} was also a deliberate choice. GraphQL would have been defensible for highly variable client-side queries, but this project has a relatively stable set of resources and workflows. REST made route semantics, status codes, authentication guards, and Swagger-based documentation more direct \cite{openapi}.

\section{Implementation}
\subsection{Core Data, CRUD, and Resource Design}
To satisfy the coursework requirement for database-backed CRUD, the backend exposes full RESTful CRUD for the \texttt{Item} model:
\begin{itemize}
  \item \texttt{POST /api/items} creates an item,
  \item \texttt{GET /api/items} and \texttt{GET /api/items/:id} read user items,
  \item \texttt{PATCH /api/items/:id} updates an item,
  \item \texttt{DELETE /api/items/:id} deletes an item.
\end{itemize}
These operations are backed by PostgreSQL through Prisma and return conventional JSON status-code responses.

The \texttt{Item} model was the most appropriate candidate for full CRUD because it is a stable resource with clear lifecycle operations and conventional REST semantics. In the submitted app, however, CRUD is not the main user-facing story; the visible experience is recommendation-led, while these resources mainly support the assessed API surface and stored recommendation structures. The listings subsystem was also extended to support a fuller lifecycle: listings can be updated or deleted by their owner, and deleting a listing releases the linked item back to the \texttt{AVAILABLE} state instead of leaving it trapped in a reserved marketplace state.

\subsection{Recommendation Pipeline}
The recommendation service combines two primary signals. First, climate suitability measures whether a suggested look is likely to be comfortable for the chosen location and time. Second, trend relevance applies a bounded boost or penalty from recent magazine-derived fashion signals gathered from curated feeds. The recommendation endpoint remains publicly usable, while authenticated requests unlock saved recommendations, account settings, and more tailored chat assistance.

This design is especially useful in the target use case. A user getting ready quickly may know that the day is ``cold'' or ``rainy'', but not whether the weather will change by the time they leave. By consolidating these signals into one recommendation, the system reduces fragmented decision-making and supports comfort and self-presentation. The more original aspect of the design is that climate, trend, and preference signals are not merely displayed side by side; they are fused into one bounded scoring pipeline whose contribution can still be explained and defended.

\subsection{Data Flow Through the System}
The operational flow begins when the user sends a recommendation request, optionally after logging in. The client collects a location and time, the API resolves weather context, and trend services contribute recent magazine-style signals. A saved account location can prefill the form, while blank settings intentionally do not force an autofill. After scoring candidate looks, the API returns a JSON response containing the selected result, and logged-in users can store that recommendation with its weather, place, time, stylist blurb, and a small deduped set of inspiration image URLs for later review. A separate chat-assistant flow can then turn the same context into more conversational styling advice. In the final frontend, that assistant is deliberately streamlined around preset activity selections rather than an unrestricted compose box, which reduces noisy prompts and keeps the guidance more consistent during demonstration.

\subsection{Security, Error Handling, and Operational Choices}
Security was addressed at multiple layers.

\textbf{Authentication.} The API uses JWT-based authentication with guarded routes. This provides a simple stateless security model for a coursework API and allows protected ownership checks without server-side session storage \cite{jwt}. The account area also supports authenticated password change with current-password verification before a new bcrypt hash is stored. JWT secrets are treated as required environment configuration rather than silently falling back to hard-coded defaults.

\textbf{Authorisation.} Ownership checks were added to CRUD operations and saved recommendations so a user can only read, update, or delete their own records. This prevents horizontal privilege issues and ensures the API does not expose another user's private data.

\textbf{Validation and error handling.} DTO validation is enforced in NestJS using class-validator and a global validation pipe. Explicit \texttt{401}, \texttt{403}, \texttt{404}, and \texttt{409} responses make failure states predictable instead of ambiguous. This is particularly important for ownership-controlled endpoints, where incorrect status behaviour can hide security mistakes during testing.

\textbf{Proxy policy.} The image proxy used for Pinterest inspiration was tightened so it only accepts HTTPS requests to approved Pinterest image hosts, applies a timeout, and verifies that the upstream response is an image. This reduces the risk of turning a convenience proxy into a general server-side fetch endpoint.

\textbf{Operational trade-off.} JWT was selected because it is simple and stateless, but token revocation is weaker than in a full server-side session model. For this coursework-scale system, simplicity and ease of integration outweighed that limitation.

\subsection{Documentation, Version Control, and Deployment Readiness}
The project is documented in two complementary ways: Swagger-based API documentation for endpoint use, and this report for architectural rationale, testing, and limitations. The final submission was tightened so these artifacts describe the same recommendation-led system rather than contradictory scopes.

Version control was managed through GitHub using named branches and incremental commits for major features such as recommendation refinement, authentication hardening, chat-assistant integration, trend-driven interface changes, and documentation updates. This supported both experimentation and a cleaner final assessed surface.

The system is demonstrable through local execution. Backend and frontend builds run successfully, the API exposes JSON responses over HTTP, and a Swagger interface is available for interactive testing. These factors satisfy the coursework requirement that the system be runnable and inspectable rather than merely described.

\section{Evaluation}
\subsection{Usefulness and Design Value}
The system is useful because it reduces cognitive load during a time-constrained daily task. Users may know the weather in general terms but still miss forecast detail at the exact time they are going out, and may only have vague awareness of current fashion coverage. Style Forecast turns those fragmented signals into one actionable suggestion.

\subsection{Testing and Validation}
Evaluation was carried out in four ways:
\begin{enumerate}
  \item backend and frontend production builds to confirm deployable integrity,
  \item endpoint validation through Swagger and direct HTTP requests,
  \item automated unit tests for recommendation and account logic,
  \item end-to-end flow checks across registration, authenticated recommendation, saved recommendations, password change, and chat-assisted guidance.
\end{enumerate}

The evaluation is qualitative rather than benchmark-driven: the main question is whether the recommender gives plausible, explainable, and secure recommendations under realistic usage conditions. In addition to unit tests, the assessed backend includes integration coverage for authenticated recommendation behaviour, saved recommendation flows, and chat-assisted guidance.

\subsection{Challenges, Lessons, and Future Work}
The most difficult implementation issue was timezone correctness. Weather relevance depends on when and where the user intends to wear an outfit, so naive datetime handling produces incorrect recommendations. This also affected account settings, because the interface needed to display an obvious user-facing timezone label while still storing a robust underlying zone identifier. A second challenge was external data reliability, so the system had to fail clearly rather than silently.

The main lesson from the project was that explainability matters as much as raw feature count. A smaller system with clear logic, secure defaults, and a strong end-to-end flow is more defensible than a larger but less coherent implementation. Architectural separation between controller, service, and persistence layers also proved worthwhile when CRUD, JWT refresh, and documentation changes had to be introduced without destabilising unrelated behaviour.

The current trend sentiment model is lexical and can miss nuance. Personalisation is heuristic rather than learned from large-scale data, and automated coverage remains narrower than production standards. Future work would therefore focus on broader automated testing, scheduled trend ingestion, constrained learning-to-rank, and richer experimentation with explanation-preserving ranking methods rather than opaque generation-heavy recommenders.

\section{Conclusion}
Style Forecast was designed to bridge a genuine everyday knowledge gap: what to wear when time is limited and contextual information is fragmented. The final architecture reflects deliberate choices about maintainability, security, explainability, documentation quality, and practical usefulness. The project therefore demonstrates not only a working API, but a defensible design process backed by version control, testing, and structured technical reasoning.
