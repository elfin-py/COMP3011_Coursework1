\section{Project Overview}
Style Forecast is a full-stack recommendation system for clothing decisions under real-world uncertainty. The core objective is to recommend outfits that are not only stylistically relevant, but also practical for weather conditions and adaptive to user preferences over time. The system is API-first: the backend exposes reusable endpoints and the frontend acts as one client of that API.

The design intentionally prioritises three qualities:
\begin{enumerate}
  \item \textbf{Explainability}: recommendations are produced from interpretable score components.
  \item \textbf{Adaptivity}: the system should improve as user feedback accumulates.
  \item \textbf{Operational simplicity}: architecture should remain easy to run, test, and demonstrate locally.
\end{enumerate}

\section{Technology Stack and Personal Rationale}
\subsection{Programming Language}
I chose \textbf{TypeScript} for both backend and frontend to reduce context switching and avoid common interface mismatches between layers. Sharing a single typed language improved development speed and confidence during refactoring.

\subsection{Backend Framework}
I selected \textbf{NestJS} because its module-oriented structure aligns well with this domain. Separate modules (\texttt{auth}, \texttt{items}, \texttt{outfits}, \texttt{climate}, \texttt{recommendation}, \texttt{trends}) make responsibilities explicit and keep business logic concentrated in services rather than controllers. This separation was useful when iterating on scoring logic without destabilising HTTP routes.

\subsection{Database and ORM}
I used \textbf{PostgreSQL} with \textbf{Prisma}. An SQL database is appropriate for this project because the domain is strongly relational:
\begin{itemize}
  \item users own many items and outfits,
  \item outfits contain many items,
  \item feedback and usage logs reference users and targets,
  \item climate snapshots and trend signals are queried by time windows.
\end{itemize}
PostgreSQL provided robust relational constraints and predictable query behaviour. Prisma added migration control and type-safe queries, reducing runtime errors in data access code.

\subsection{Frontend}
I used \textbf{React + Vite}. React gave component-level control for an interactive recommendation flow; Vite kept build and reload times low during frequent iteration.

\section{Architecture and Key Design Choices}
\subsection{High-Level Structure}
The architecture has three layers:
\begin{itemize}
  \item \textbf{Presentation layer}: React client for user interaction.
  \item \textbf{Application layer}: NestJS controllers and services for API behaviour and domain logic.
  \item \textbf{Data layer}: PostgreSQL accessed through Prisma models and migrations.
\end{itemize}

\subsection{Domain Modelling}
The data model includes \texttt{User}, \texttt{Profile}, \texttt{Item}, \texttt{Outfit}, \texttt{OutfitUsage}, \texttt{ClimateSnapshot}, \texttt{SocialTrend}, and \texttt{Feedback}. The design separates stable identity data (user/profile) from event data (usage, feedback, climate, trends), which simplifies analytics and recency-based querying.

\subsection{API Style and Error Semantics}
I implemented a RESTful API with JSON payloads and conventional status codes:
\begin{itemize}
  \item \texttt{201} for created resources,
  \item \texttt{200} for successful reads/updates,
  \item \texttt{204} for successful deletion,
  \item \texttt{400/401/403/404/409} for validation, auth, ownership, missing resources, and conflicts.
\end{itemize}
This choice improves predictability for clients and keeps behaviour aligned with common industry practice.

\section{CRUD Coverage and Demonstrability}
To satisfy core API requirements, the \texttt{Item} model implements full CRUD against PostgreSQL:
\begin{itemize}
  \item \textbf{Create}: \texttt{POST /api/items}
  \item \textbf{Read}: \texttt{GET /api/items}, \texttt{GET /api/items/:id}
  \item \textbf{Update}: \texttt{PATCH /api/items/:id}
  \item \textbf{Delete}: \texttt{DELETE /api/items/:id}
\end{itemize}
Ownership checks are enforced on read/update/delete operations to prevent cross-user access. The application is demonstrable by local execution with Docker-backed PostgreSQL and standard npm scripts for build and run.

\section{Recommendation Approach}
The recommendation pipeline blends:
\begin{itemize}
  \item weather suitability (temperature, precipitation, wind),
  \item trend influence (recent sentiment/volume signals),
  \item user adaptation (feedback-informed boost).
\end{itemize}
I intentionally chose a weighted heuristic design over opaque end-to-end models. The trade-off is lower model complexity, but much higher interpretability and easier debugging during assessment demonstrations.

\section{Testing Approach and Validation}
My testing strategy combined:
\begin{enumerate}
  \item \textbf{Build validation}: backend and frontend production builds as a baseline integrity check.
  \item \textbf{Endpoint verification}: manual API checks through Swagger and HTTP requests.
  \item \textbf{Flow checks}: end-to-end user paths (auth, item creation, outfit recommendation, feedback).
\end{enumerate}
This provided functional confidence for coursework delivery. The main remaining gap is broader automated coverage (especially edge-case unit tests and integration assertions for external API failures).

\section{Challenges and Lessons Learned}
\subsection{Timezone Correctness}
The most persistent issue was time interpretation across locations. Converting user-selected local datetime values to consistent server-side query time was essential to avoid incorrect forecast matching.

\subsection{External Data Reliability}
Weather and trend endpoints introduce variability in latency and availability. Designing graceful failure paths and clear error responses was necessary for stable behaviour.

\subsection{Scope Control}
A practical lesson was to keep feature growth aligned with assessment goals. Prioritising one model with complete CRUD, robust auth, and clear docs was more valuable than adding broad but shallow features.

\section{Limitations and Future Improvements}
\subsection{Current Limitations}
\begin{itemize}
  \item Trend sentiment extraction is lexical and can misread context.
  \item Personalisation is heuristic rather than learned from large interaction histories.
  \item Automated test coverage is currently narrower than production standards.
\end{itemize}

\subsection{Future Development}
\begin{itemize}
  \item Expand automated tests (unit, integration, and regression suites).
  \item Introduce scheduled/background ingestion jobs for trend processing.
  \item Evolve ranking from fixed weights toward a constrained learning-to-rank approach while preserving explainability.
  \item Add observability and rate-limiting for more robust deployment.
\end{itemize}

\section{Conclusion}
The project demonstrates a practical API-centric system that combines SQL-backed CRUD functionality with adaptive recommendation logic. The key design decisions were driven by maintainability, explainability, and demonstrability under coursework constraints. The resulting architecture is a solid base for future iteration toward stronger personalisation and production-grade reliability.
